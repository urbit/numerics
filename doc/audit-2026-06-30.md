# Urbit Numerics Angel Audit — 2026-06-30

Methodology: five parallel angel agents (math, lagoon, saloon, unum+cplx+fixp+twoc,
READMEs) plus a synthesis pass.  ~465k tokens, 99 tool calls.  "Confirmed" means
source-traced or bit-verified against IEEE/SoftFloat/SoftPosit references; ship
verification still needed for the three H-level code bugs (H1, H4, H5).

**Total findings (deduplicated):** 55 — 11 high, 27 medium, 17 low

**Jet compliance note (2026-06-30):** Jets exist for `%i754` operations in
`libmath` (math doors) and `lagoon` (array ops via SoftBLAS).  `%cplx`,
`%unum`, `%fixp`, `%int2` always fall through to Hoon — no jet drift risk for
those kinds.  Two code bugs interact with jets:

- **H1** (`+huge` wrong bit pattern): `+huge` is not referenced by any jet;
  the C jets use an internal `_RS_PINF` constant entirely.  Bug is Hoon-only.
- **H4** (`++range` @rh shape): `u3wi_la_range` correctly sets
  `x_shape = u3nc(n, u3_nul)` for all four bloq cases at line 3577 of
  `lagoon.c`.  The **C jet is correct**; the Hoon fallback is missing the
  `=.  shape.meta  ~[(lent bad)]` update for bloq 4.  When the jet fires the
  caller gets the right shape; when it falls through to Hoon (unjetted ship)
  the shape is wrong.  Fix the Hoon to match the jet.
- **H5** (`++mod` %cplx): `u3wi_la_mod` handles only `c3__i754`; the default
  returns `u3_none`, falling through to Hoon which then fires `!!`.  Jet and
  Hoon agree on the crash; the fix is to add a diagnostic message in Hoon and
  document it.

## Section 1 — `libmath/desk/lib/math.hoon`

### HIGH

**H1 — `+huge` in `++rs` encodes `+Infinity`, not the largest finite float**
- Location: `++rs`, arm `+huge`, line ~105
- Kind: wrong-claim (code bug — wrong bit pattern)
- Detail: `` `@rs`0x7f80.0000 `` is IEEE 754 binary32 +Inf.  The doccord,
  inline comment, and example all say `3.40282346638528859812e+38` — the
  largest *finite* single, `0x7f7f.ffff`.  `++rd` and `++rh` have the correct
  bit patterns.
- Fix: `` ++  huge  `@rs`0x7f7f.ffff  ::  3.40282346638528859812e+38 ``
- Jet impact: none (`+huge` is not jetted; `_RS_PINF` in `math.c` is
  independent).

**H2 — `+round` doccord examples are off by one**
- Location: `++rs` arm `+round` lines ~1027–1051; `++rd` arm `+round` ~2197–2221
- Kind: wrong-claim
- Detail: For `x=1.11`, `n=1`: `scaling = 10^(1-0-1) = 1`, `round(1.11)=1`,
  result `1.0` not `1.1`.  Every listed example shows the result for `n+1`.
  "decimal places" in the description also conflicts with "significant digits"
  in the formula.
- Fix: Correct all three examples to `n=1→1`, `n=2→1.1`, `n=3→1.11` and
  change "decimal places" to "significant digits (n=1 means 1 sig fig)."

**H3 — `@rq` `+tan` `pi` example shows `~-2.185` (≈ `tan(2.0)`) instead of near-zero**
- Location: `++rq`, arm `+tan`, line ~3891
- Kind: wrong-claim
- Detail: Copy-paste error from `@rq` `tan(2)` result.  All other doors show
  small values near 0.
- Fix: Compute on ship: `(~(tan rq:math [%n .~~~1e-20]) pi:rq:math)` and
  update.

## Section 2 — `lagoon/desk/lib/lagoon.hoon`

### HIGH

**H4 — `++range` for `@rh` (bloq 4) does not update `shape.meta`**
- Location: lines ~732–738 (bloq `%4` branch)
- Kind: wrong-claim (code bug — Hoon/jet divergence)
- Detail: Bloq 5/6/7 branches all execute `=.  shape.meta  ~[(lent bad)]`
  before returning.  The bloq 4 branch returns `[meta (flop bad)]` directly,
  propagating the caller-supplied shape instead of the generated length.  The
  C jet `u3wi_la_range` (lagoon.c line 3577) correctly sets
  `x_shape = u3nc(n, u3_nul)` for bloq 4 — so **the jet is correct and the
  Hoon is wrong**.  Callers on a jetted ship get the right shape; callers on an
  unjetted ship get corrupted metadata.
- Fix: Add `=.  shape.meta  ~[(lent bad)]` before `[meta (flop bad)]` in the
  bloq 4 branch.  Hoon then matches the jet.

**H5 — `++mod` doccord claims "NaN on a zero divisor" for all kinds; `%cplx` has no dispatch and crashes silently**
- Location: `++mod` doccord lines ~1341–1344; `++fun-scalar` `%cplx` block
  lines ~1783–1836
- Kind: wrong-claim + missing crash message
- Detail: "NaN" is accurate only for `%i754`.  `%cplx` hits the `?+` wildcard
  and fires bare `!!`.  The C jet returns `u3_none` for non-%i754 kinds,
  falling through to Hoon — so the crash path is always hit.
- Fix: Add `~|(%cplx-mod-unsupported !!)` as the `%cplx` fallback; rewrite
  doccord with per-kind behavior.
